Framework · 决策框架

RISD决策框架：四问判断法

四个问题，快速判断是否值得为某个需求编写Skill

R
重复性
这个任务会反复执行吗？
一次性任务不值得写Skill
→
I
信息充分性
所需信息是否完整可获取？
信息不全则无法标准化
→
S
可拆分性
能否拆分为清晰步骤？
不可拆分的模糊任务不适合
→
D
确定性
是否有明确的判断标准？
标准模糊则质量不可控
RISD法则：重复且信息充分且可拆分且确定的任务，是Skill的最佳候选人。

Method · 步骤详解

Step 1-2：目标锁定与输入边界

先定边界再动手——最被低估的两步

Step 1 · Plan
用一句话说清目标：
• 这个Skill要解决什么问题？
• 成功的标准是什么？
• 一句话总结核心价值
"目标越清晰，Skill越锋利"
Step 2 · Requirements
明确输入边界：
• 需要哪些输入信息？
• 输入的格式和约束？
• 缺失信息时的降级策略

Method · 步骤详解

Step 3-4：步骤拆解与能力定义

确定"做什么"和"能做什么"

Step 3 · Operations
把任务拆成原子步骤：
• 每一步只做一件事
• 步骤之间有明确衔接
• Agent擅长什么交给Agent
  LLM擅长什么交给LLM
Step 4 · Capabilities
定义能力边界：
• 允许做什么（白名单）
• 禁止做什么（黑名单）
• 关键是"不做什么"
• 禁止清单比许可清单更重要

Method · 步骤详解

Step 5-7：验收→编写→调试

从标准到代码到验证——最后一公里

Step 5 · Evaluation
定义验收标准：
• 成功的输出长什么样
• 失败的边界在哪里
• 可量化的检查点
Step 6 · Script
编写结构化脚本：
• 先搭骨架再填肉
• 分步骤、带注释
• 每一步有独立校验
Step 7 · Sanity
测试与迭代：
• 正案例测试→正常输入
• 负案例测试→异常输入
• 边角案例测试→极端输入

Principle · 接口与容错

接口与容错原则

面向未来的设计——经得起变化、容得下错误

① 向前兼容
开放扩展，不破坏旧用法
② 默认即正确
零配置开箱即用
③ 单一职责
一个Skill只做一件事
④ 版本稳定
接口不随意变更
"容错不是备选项，是基本功。一个不能容错的Skill，用户用一次就弃。"

Principle · 测试与性能

测试与性能原则

可验证的质量——每一步都可以被检查

⑤ 用例先行
先写测试用例再写代码
⑥ 步步校验

"Skill的质量不靠感觉，靠校验。每个步骤必须有可验证的输出标准。"

Principle · AI调用稳定性

AI调用稳定性原则

让AI稳定输出——而不是"有时候好，有时候差"

⑨ 描述带攻击性
指令要强硬明确，不留解读空间
⑩ 边界清晰
输入输出范围一目了然
⑪ 白名单策略
允许什么、禁止什么，清单化
⑫ 样例驱动
提供好/坏样例，AI需要对比参照
"AI不是不听话，是你没说清楚。描述越有攻击性，输出越稳定。"
⑬ 强制结构输出
用格式约束内容，JSON/YAML结构化输出

Principle · 约束与迭代

约束与迭代原则

Skill的生命力在于持续进化

⑭ 幻觉控制
关键步骤加校验，AI编造数据不可信
⑮ 反馈闭环

⑯ 迭代维护
Skill是活的，不是写完就完
"一个Skill的生命周期：写→用→反馈→改→再写。循环越快，质量越高。"
核心心法：一句话 = 一个Skill，少即是多。

Case 01 · bug-fixer

bug-fixer：系统性Bug修复专家 v2

从"一刀切"到"分诊 + 7条专用路径 + 全局原则"的架构升级

核心架构
分诊层：问题类型自动分流
7条修复路径：A崩溃 B数据 B+需求 D性能 E并发 F环境 G构建
5项全局能力：三次法则、橡皮鸭复述、假设验证、结构化输入、经验沉淀
7条全局原则
① 不打补丁——A/B双重修复，现象+根因
② 先见红——无证据不修改，测试先行
③ 系统性——修一个、扫一类
④ 两方案——设计选择必须对比
⑤ 自我批判——每步检查是否过度工程
⑥ 先沟通——不确定时先问再动手
⑦ 经验沉淀——YAML格式结构化记录
bug-fixer 完美体现了PROCESS方法论和16条成功原则的落地——分诊即Plan，路径即Operations，原则即Constraint。

Case 02 · refactor-22

refactor-22：22种代码坏味道自动识别与重构

对标 Martin Fowler《重构》第二版，全库扫描 → 优先级排序 → 逐项修复

F1-F7 高优先级
重复代码 → Extract Method
过长函数 → Extract / Replace Temp
过大的类 → Extract Class
过长参数列 → Parameter Object
霰弹式修改 → Move Method
F8-F15 中优先级
数据泥团 → Extract Class
基本类型偏执 → Replace with Object
Switch语句 → 多态替换
拒绝的遗赠 → Replace Subclass
中间人 → Remove Middle Man
F16-F22 低优先级
全局数据 → Encapsulate Variable
可变数据 → Split Variable
循环语句 → Replace with Pipeline
冗赘的元素 → Inline Class
夸夸其谈通用性 → Collapse Hierarchy
refactor-22 是一个典型的知识密集型Skill：22种坏味道 → 22条标准策略 → 3阶段执行流程。知识越结构化，AI执行越可靠。
案例：refactor-22
