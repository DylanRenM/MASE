第七讲：综合案例 —— refactor-22 代码重构专家

refactor-22 是什么？

对标 Martin Fowler《重构》第二版，自动识别并修复 22 种代码坏味道的智能重构工具。

🔴 F1-F7 高优先级
  • 重复代码 → Extract Method
  • 过长函数 → Extract
  • 过大的类 → Extract Class
  • 过长参数列 → Parameter Object
  • 霰弹式修改 → Move Method
🟠 F8-F15 中优先级
  • 数据泥团 → Extract Class
  • 基本类型偏执 → Replace with Object
  • Switch语句 → 多态替换
  • 拒绝的遗赠 → Replace Subclass
  • 中间人 → Remove Middle Man
🟢 F16-F22 低优先级
  • 全局数据 → Encapsulate Variable
  • 可变数据 → Split Variable
  • 循环语句 → Replace with Pipeline
  • 冗赘的元素 → Inline Class
  • 夸夸其谈通用性 → Collapse Hierarchy

三层架构设计

01
检测层 | Detection Layer
自动识别22种代码坏味道，基于静态分析和模式匹配，生成问题清单和优先级排序
02
策略层 | Strategy Layer
每种坏味道对应一条标准修复策略，从 Martin Fowler《重构》中提炼的经典解决方案
03
执行层 | Execution Layer
3阶段执行流程：分析 → 重构 → 验证，确保每一步都可追溯、可回滚

用 RISD 框架判断：值得写 Skill 吗？

✅ R - Repeatability（重复性）
代码重构是日常开发的常规工作，每天都可能需要对代码进行质量检查和优化
✅ I - Information（信息充分性）
输入是源代码文件，信息明确；22种坏味道有明确的检测规则
✅ S - Stability（稳定性）
检测规则和修复策略是固定的，基于经典重构理论，不会频繁变化
✅ D - Divisibility（可拆分性）
检测用脚本，策略用规则引擎，执行用Agent，可清晰分工

结论：四个条件都满足 → 非常适合写成 Skill！

用 PROCESS 框架构建 refactor-22

  • P - Plan：明确目标是"自动检测并修复22种代码坏味道"
  • R - Requirements：输入是源代码目录路径，输出是修复后的代码和重构报告
  • O - Operations：检测层 → 策略层 → 执行层，三步流程
  • C - Capabilities：需要文件读写、代码解析、命令执行权限
  • E - Evaluation：输出格式为Markdown报告，包含问题清单和修复建议
  • S - Script：编写YAML格式的Skill定义
  • S - Sanity：测试用例覆盖所有22种坏味道的检测和修复

为什么用 Skill 而不是 Prompt？

❌ 使用 Prompt 的问题
每次都要描述22种坏味道的检测规则，冗长且容易出错；无法保证输出格式一致；无法执行实际的代码修改操作
✅ 使用 Skill 的优势
一次配置，多次复用；规则固化，输出一致；可以调用脚本执行实际重构；有明确的权限控制和错误处理

为什么用 Skill 而不是纯程序？

❌ 纯程序的局限性
规则硬编码，修改困难；无法理解上下文语义；面对复杂代码模式容易误判；缺乏灵活性
✅ Skill 的智能优势
LLM理解代码语义，检测更准确；规则可以动态调整；支持上下文感知的重构建议；灵活应对各种代码风格

16条成功原则在refactor-22中的体现

  • 见名知意：参数命名清晰（file_path, refactor_type）
  • 语义化版本：v1.0 → v1.1（新增检测规则）→ v2.0（新增语言支持）
  • 用例先行：每种坏味道都有测试用例
  • 步步校验：检测结果要验证，修复后要验证
  • 固化责任：确定性检测用脚本，语义理解用LLM
  • 白名单优于黑名单：明确允许修复的范围
  • 样例胜过描述：每种重构都有示例
  • 反馈闭环：收集误报/漏报，迭代优化规则
