# 流程图评审 (Flowchart Review)

将代码逻辑或设计文档图形化为需求级实现流程图（Mermaid 格式），人在流程图层面做设计评审，标注反馈后驱动代码和设计的修复。

## 适用场景

- Verify 阶段：对照项目的设计文档（`openspec/changes/{name}/detailed-design.md` + `architecture.md`）检查代码实现逻辑是否偏离设计
- Build 阶段：复杂模块实现前先画流程图理解设计意图
- 任何需要"检查代码是否按设计走"的时刻

## 核心流程

```
判断来源 → 生成 Mermaid → 交互渲染 → 人标注 → 反馈闭环
```

## 阶段一：判断来源

- **有代码时**：读取当前会话已修改/创建的相关代码文件作为输入源
- **无代码时**：读取项目的设计文档作为输入源（`openspec/changes/{name}/detailed-design.md` 或 `architecture.md`）

## 阶段二：生成 Mermaid 流程图

### 从代码生成

```
你是一个需求分析师。请阅读以下代码，生成该功能的需求级实现流程图。

要求：
1. 输出 Mermaid flowchart TD 格式
2. 每个节点使用唯一且有意义的 ID（如 upload_start, validate_type）
3. 覆盖：正常路径、异常分支、边界条件
4. 节点类型用形状区分：
   - 方框 [文本] → 动作/操作
   - 菱形 {文本} → 判断/分支
   - 圆角 ([文本]) → 开始/结束
5. 异常分支必须标注触发条件（如"|超时|"、"|校验失败|"）
6. 只输出 Mermaid 代码块，不要额外解释

代码：
{代码文件内容}
```

### 从设计文档生成

```
你是一个需求分析师。请阅读以下设计文档，生成该功能的需求级实现流程图。

要求：
1. 输出 Mermaid flowchart TD 格式
2. 每个节点使用唯一且有意义的 ID
3. 覆盖：正常路径、异常分支、边界条件
4. 节点类型用形状区分（方框=动作，菱形=判断，圆角=端点）
5. 设计文档中未明确的异常分支，请基于经验合理推断并标注"[推断]"
6. 只输出 Mermaid 代码块，不要额外解释

设计文档：
{项目设计文档内容}
```

## 阶段三：交互评审与反馈

Mermaid 代码渲染为交互式组件：

1. 用户可悬停节点并添加标注（问题/建议/疑问 + 严重度）
2. 节点右上角显示标注数量徽章
3. 右侧面板列出所有标注
4. 工具栏提供：缩放、显示/隐藏标注、通过、修正

## 反馈闭环

用户点击"修正"后，汇总未解决标注，注入 LLM 上下文：

```
你收到了以下流程图评审反馈。请根据反馈修改对应代码/设计。

标注反馈：
- [severity] 节点 {nodeId}: "comment"
...

要求：
1. 只修改与标注相关的代码/设计，不改无关内容
2. 保持现有代码风格
3. 修改后输出完整文件内容
```

修订完成后自动重新生成 Mermaid 流程图，已解决标注标记为 resolved。

## MASE 定位

- **所属阶段**: Verify（可选工具）
- **与 Agent 4 的关系**: 辅助工具，非门禁必做项
- **典型调用**: "画个流程图看看这段逻辑对不对" / "flowchart review 这个模块"
