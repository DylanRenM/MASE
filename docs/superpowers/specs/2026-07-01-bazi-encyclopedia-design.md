# 八字百科全书 (BaziEncyclopedia) 设计文档

> 状态: 已确认 | 日期: 2026-07-01

## 1. 项目概述

将上百本八字命理书籍自动融合为一本结构化的百科全书，支持增量添加书籍。

### 核心目标
- 输入：100+ 本八字电子书（EPUB/PDF/TXT/扫描件）
- 输出：按 [流派 × 专题] 二维结构组织的百科全书 + 可检索命例数据库
- 交互：Web 界面，左栏书籍管理 + 右栏百科浏览（目录树 + 内容区）

### 流派分类
旺衰派、格局派、盲派、神煞、调候、病药、阴阳

### 专题分类
婚姻、健康、寿命、六亲、财运、事业、学业、性格、用神、大运、流年、基础理论

## 2. 技术架构

### 2.1 总览

```
输入(N本书) → 阶段1:解析分块 → 阶段2:向量化+分类 → 阶段3:聚类去重 → 阶段4:百科编排 → 输出(Web/EPUB/PDF)
```

### 2.2 技术栈

| 组件 | 选型 | 说明 |
|------|------|------|
| Embedding 模型 | bge-large-zh-v1.5 | 本地运行，768维中文向量 |
| 向量数据库 | Qdrant | 嵌入式部署，支持过滤查询 |
| LLM | Qwen3-32B（Ollama/vLLM） | 本地部署，128K上下文 |
| 元数据存储 | SQLite | 书源状态、进度管理 |
| 百科内容 | JSON | 结构化存储，易于前端消费 |
| 前端 | 单HTML + 原生JS/CSS | 无依赖，浏览器直接打开 |
| 后端服务 | Python (FastAPI) | 管道调度 + API 接口 |

## 3. 四阶段管道设计

### 3.1 阶段1：批量解析与统一分块

**输入格式处理：**

| 格式 | 处理方式 |
|------|----------|
| TXT | 直接读取，编码检测 |
| EPUB | 解压解析 XHTML |
| PDF（文字型） | 直接文本提取 |
| PDF（扫描版）/ 古籍 | OCR (PaddleOCR) + 后处理纠错 |

**分块参数：**
- 块大小：300-500字
- 分块原则：按段落边界切割，优先保持语义完整
- 相邻块：50字重叠窗口

**元信息：**
- source_book, author, chapter_path, block_id

**估算：** 100本 × 20万字 ≈ 2000万字 → 约50,000块

### 3.2 阶段2：全量向量化 + 初步分类

**流程：**
1. 全部块 → bge-large-zh-v1.5 → 768维向量 → Qdrant
2. 每个块送本地千问做三分类（一次 Prompt 输出）：
   - genre_tags: 流派标签（可多选）
   - topic_tags: 专题标签（可多选）
   - content_type: theory / case_study / mixed
   - summary: 10字概要

**性能估算：** 50,000块，Qwen3-32B 本地推理约 2-3 小时

### 3.3 阶段3：聚类去重（核心）

**第一轮 — 向量聚类 (DBSCAN)：**
- 在 Qdrant 内按 [流派×专题] 分组
- 组内 DBSCAN 聚类，cosine_similarity ≥ 0.85 归入同一"知识簇"
- 离群点（噪音）自动标记为独有内容，直接保留

**第二轮 — LLM 精判：**
- 只对簇内 ≥2 块的"知识簇"送 LLM
- 判定四类：
  - `duplicate` → 保留最优版本
  - `complementary` → 保留全部，标记待融合
  - `conflicting` → 保留全部，标记"流派分歧"
  - `distinct` → 拆分为独立条目

**估算：** 约15,000候选对，本地千问约1-2小时

### 3.4 阶段4：百科全书编排

**条目结构：**
```
# [专题名] — [流派名]视角
## 核心观点（融合互补块）
## 各家论述（带出处引用）
## 流派分歧（如有冲突）
## 经典命例
## 参见（交叉引用）
```

**命例数据库：** 独立子系统，按专题/流派/八字特征三维检索

## 5. 数据存储设计

### SQLite 表结构

```sql
-- 书籍管理
CREATE TABLE books (
    id TEXT PRIMARY KEY,
    filename TEXT NOT NULL,
    title TEXT,
    author TEXT,
    format TEXT,           -- epub/pdf/txt
    status TEXT DEFAULT 'pending',  -- pending/processing/done/failed
    progress REAL DEFAULT 0.0,
    block_count INTEGER DEFAULT 0,
    created_at TIMESTAMP,
    merged_at TIMESTAMP
);

-- 处理管道日志
CREATE TABLE pipeline_logs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    book_id TEXT REFERENCES books(id),
    stage TEXT,            -- parse/embed/classify/cluster/merge
    status TEXT,           -- running/done/failed
    message TEXT,
    created_at TIMESTAMP
);
```

### Qdrant Collection

- collection: `bazi_knowledge`
- vector: 768维
- payload: {book_id, genre_tags, topic_tags, content_type, chapter, block_text}

### 百科内容 JSON

```json
{
  "encyclopedia": {
    "旺衰派": {
      "婚姻": {
        "core_view": "...",
        "sources": [{"book": "...", "quote": "..."}],
        "conflicts": [{"view": "...", "source": "..."}],
        "cases": ["case_id_1", "case_id_2"],
        "see_also": ["格局派.婚姻", "调候.女命"]
      }
    }
  }
```

## 6. 关键技术决策

| 决策 | 理由 |
|------|------|
| 本地部署全部模型 | 百本书Token量巨大，API成本不可接受 |
| DBSCAN 而非 K-Means | 无法预设知识点数量，需自动识别离群独特内容 |
| 向量聚类 + LLM精判 两轮去重 | 纯向量误判率高（命理术语密集），纯LLM太慢 |
| 单HTML前端 | 零部署，本地浏览器即可使用 |
| 命例数据库独立但与百科互跳 | 命例分析模式和知识查阅模式不同 |

## 7. 风险与边界

### 已知风险
1. **扫描版PDF OCR准确率** — 古籍繁体、竖排、无标点是OCR难点，可能需要人工校对
2. **流派标签模糊** — 有些书籍跨流派，分类标签可能不准，需LLM多标签支持
3. **去重误判** — 同一概念不同流派表述差异大，LLM可能误判为distinct而非complementary
4. **输出质量控制** — 最终百科的学术可靠性依赖LLM输出质量+人工审校

### 不做的事情（范围边界）
- ❌ 不提供在线服务/SaaS
- ❌ 不支持用户注册/多用户
- ❌ 不做移动端App
- ❌ 不自动从网上下载书籍
- ❌ 不保证100%准确的去重（人工可审校修正）
