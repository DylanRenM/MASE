# MASE 跨语言通用编码规范

> **加载策略**: 按需加载。编码时由 Agent 3 注入上下文，不常驻内存。
> **语言特定规范**: Design L2 技术栈确定后，Agent 3 在 `openspec/changes/{name}/coding-standards.md` 中追加语言特定规则。

## 1. 命名规范

| 规则 | 说明 |
|------|------|
| 有意义 | 名称必须表达意图，`getUserById` 而非 `get` |
| 可拼读 | 名称应能读出来，`createdAt` 而非 `crtd_at` |
| 不用缩写 | 除非是通用缩写（`id`, `url`, `json`, `xml`, `html`） |
| 布尔值前缀 | `is_`, `has_`, `can_`, `should_` |
| 集合用复数 | `users` 而非 `user_list` |
| 不重载含义 | 同一名称在同一上下文只表达一个含义 |

## 2. 硬编码禁令

| 禁止 | 替代方式 |
|------|---------|
| 魔法数字 | 命名常量，如 `MAX_RETRY_COUNT = 3` |
| 魔法字符串 | 配置项或枚举，如 `STATUS_ACTIVE = "active"` |
| 内联路径 | 集中定义在 `config/` |
| 内联 URL | 环境变量或配置文件 |

## 3. 函数规范

| 规则 | 阈值 |
|------|------|
| 单一职责 | 一个函数只做一件事 |
| 行数上限 | ≤ 50 行（超过需拆分） |
| 参数上限 | ≤ 5 个（超过用对象/结构体封装） |
| 圈复杂度 | ≤ 10 |
| 无副作用 | 优先纯函数，副作用显式标注 |

## 4. 错误处理

| 规则 | 说明 |
|------|------|
| 不吞异常 | 捕获的异常必须处理或传播，不能静默忽略 |
| 不裸捕获 | 禁止 `except:` / `catch (...)`，指定具体异常类型 |
| 上下文消息 | 异常消息包含操作上下文和关键参数值 |
| 边界校验 | 所有外部输入（API 参数、文件内容、DB 数据）在入口校验 |
| 故障快速暴露 | 出错时尽早暴露，不要在不可靠状态下继续执行 |

## 5. 日志规范

| 级别 | 场景 |
|------|------|
| ERROR | 需要人工介入的故障 |
| WARN | 可自动恢复的异常，降级处理 |
| INFO | 关键业务节点（请求进入、外部调用、状态变更） |
| DEBUG | 开发调试信息，不进入生产 |

> 日志中禁止输出密码、Token、身份证号等敏感信息。

## 6. 导入规范

| 规则 | 说明 |
|------|------|
| 分组排序 | 标准库 → 第三方库 → 项目内部，组间空行分隔 |
| 显式导入 | 禁止 `import *` / 通配符导入 |
| 无循环 | 模块间不能形成导入环 |

## 7. 注释规范

| 写什么 | 不写什么 |
|--------|---------|
| 为什么这么做（Why） | 做了什么（读代码就知道） |
| 非显而易见的副作用 | 废话注释（`i++ // 自增`） |
| TODO/FIXME 需跟 issue 编号 | 过期注释（与代码实际行为不一致） |

## 8. DRY 原则

| 规则 | 说明 |
|------|------|
| 三次法则 | 同一逻辑出现第 3 次时，必须抽取为函数/方法 |
| 不提前抽象 | 不确定未来会复用时，先保持重复，等第 3 次再抽 |
| 抽取级别 | 抽到合适的抽象层级，不过早泛化 |

## 9. 语言特定规范

> 以下规则在 Design L2 技术栈确定后，由 Agent 3 补充：

```markdown
## Python 特定

- 遵循 PEP 8
- 类型注解全部函数签名
- 优先使用 dataclass / Pydantic 定义数据结构
- ...

## TypeScript 特定

- 遵循 ESLint 推荐规则
- 禁止 any（除非有充分理由并注释说明）
- 优先使用 interface 而非 type
- ...
```
