# MASE 软件设计原则

> **加载策略**: 按需加载。架构设计、设计评审时注入上下文。
> **适用阶段**: Design L2 架构设计、Agent 4 设计评审、Build 阶段重构决策。

---

## SOLID — 面向对象设计五原则

| # | 原则 | 核心含义 | MASE 评审重点 |
|---|------|---------|-------------|
| S | **单一职责** Single Responsibility | 一个类/模块只有一个变更理由 | 模块内函数是否高内聚？是否可拆？ |
| O | **开闭原则** Open/Closed | 对扩展开放，对修改关闭 | 新增功能是否需要四处改动已有代码？ |
| L | **里氏替换** Liskov Substitution | 子类必须能完全替换父类 | 继承关系是否合理？子类是否改变了父类契约？ |
| I | **接口隔离** Interface Segregation | 不应强迫客户端依赖它不使用的方法 | 接口是否过大？是否可拆为更小的角色接口？ |
| D | **依赖倒置** Dependency Inversion | 高层模块不依赖低层，都依赖抽象 | 是否依赖具体实现而非抽象？是否符合 contract.md 约束？ |

## GRASP — 通用职责分配九模式

| # | 模式 | 核心含义 | MASE 评审重点 |
|---|------|---------|-------------|
| 1 | **创建者** Creator | 谁创建对象？由包含、聚合或密切使用该对象的类创建 | 工厂函数/方法是否放在正确的模块？ |
| 2 | **信息专家** Information Expert | 谁拥有数据，谁就处理数据 | 数据和处理逻辑是否在同一模块？是否出现"贫血模型"？ |
| 3 | **低耦合** Low Coupling | 减少类之间的依赖关系 | 模块间依赖是否最小化？变更一个模块是否影响其他？ |
| 4 | **高内聚** High Cohesion | 相关职责集中在同一模块 | 模块内元素是否围绕同一核心目标？ |
| 5 | **控制器** Controller | 由专门的控制器处理系统事件 | API 路由层、中间件层次是否清晰？ |
| 6 | **多态** Polymorphism | 用多态代替条件分支 | 过长的 if/switch 是否可替换为策略模式？ |
| 7 | **纯虚构** Pure Fabrication | 为了高内聚低耦合而创造的非业务类 | 引入的中间层是否合理？是否过度？ |
| 8 | **间接性** Indirection | 通过中间对象降低直接耦合 | 是否需要适配层/代理层？解耦是否带来不必要复杂度？ |
| 9 | **受保护变化** Protected Variations | 识别不稳定点并封装 | 哪些是稳定抽象？哪些是不确定实现？是否用接口隔离了变化？ |

## 通用设计原则

| # | 原则 | 核心含义 | MASE 评审重点 |
|---|------|---------|-------------|
| 1 | **KISS** Keep It Simple, Stupid | 优先选择简单方案，不过度设计 | 方案是否过度复杂？是否有更简单的替代？ |
| 2 | **DRY** Don't Repeat Yourself | 每项知识在系统中有唯一表示 | 是否有重复逻辑？复用检查（Build Step 1）是否充分？ |
| 3 | **YAGNI** You Aren't Gonna Need It | 不预先实现未来可能需要的功能 | 是否存在"以防万一"的代码？design 是否超出 proposal 范围？ |
| 4 | **关注点分离** Separation of Concerns | 不同关注点放在不同模块 | 业务逻辑、数据访问、UI 展示是否混在一起？ |
| 5 | **最少知识** Law of Demeter | 一个对象应对其他对象有最少了解 | 方法调用链是否过深（`a.b().c().d()`）？ |
| 6 | **组合优于继承** Composition over Inheritance | 优先使用对象组合而非类继承 | 是否滥用继承？是否考虑用组合实现代码复用？ |
| 7 | **命令查询分离** CQS | 查询不产生副作用，命令不返回数据 | 读操作是否修改了状态？写操作是否返回了业务数据？ |
| 8 | **约定优于配置** Convention over Configuration | 合理的默认约定减少显式配置 | 项目结构、命名是否遵循框架约定？ |
| 9 | **防御式编程** Defensive Programming | 不信任任何外部输入 | API 入口参数是否校验？边界条件是否覆盖？ |

## 分层设计原则

| 层次 | 核心原则 | 反模式 |
|------|---------|--------|
| **API/接口层** | RESTful 一致性、幂等性、版本化管理、统一错误码 | 接口行为不一致、无版本号、错误响应格式各异 |
| **业务逻辑层** | 领域模型纯净、事务边界清晰、不变量在入口保护 | 贫血模型、事务跨越多个聚合、业务逻辑散落在 Service 层 |
| **数据/存储层** | 范式化设计、索引策略明确、迁移脚本可回滚 | N+1 查询、无索引大表扫描、不可逆迁移 |
| **UI/交互层** | 组件单一职责、状态管理单向、无障碍可访问 | 巨型组件、props 深层传递、缺失 aria 标签 |
| **安全层** | 最小权限、纵深防御、默认拒绝 | 权限校验分散、信任客户端输入、硬编码密钥 |

## 使用指南

| 阶段 | 关注的原则集合 |
|------|-------------|
| Design L2 架构设计 | SOLID + GRASP + KISS/YAGNI + 分层原则 |
| Agent 4 设计评审 | 全量对照（code-quality-controller 逐项检查） |
| Build 重构决策 | SOLID + 组合优于继承 + DRY |
| Retro 复盘 | 回顾违反的原则 → 改进措施 |
