白话 Skill

第四讲：何时需要定义为 Skill？——RISD 决策框架

一个常见的困惑

学会了 Skill 之后，你可能会想：

  • ▸ "让 AI 帮我写首诗？写个 Skill！"
  • ▸ "让 AI 解释递归？写个 Skill！"
  • ▸ "让 AI 查天气？写个 Skill！"

等等，这样很快会陷入"过度工程"的陷阱！

RISD 框架

当一个任务同时满足"重复执行、信息可得、步骤固定、可分工"四个条件时，才值得写成 Skill。

R + I + S + D = 值得写 Skill
R
Repeatability（重复性）
核心问题：这个任务会反复执行吗？
✅ 每日/每周都要做？ ✅ 特定场景下都会触发？ ❌ 只是偶尔一次？
I
Information（信息充分性）
核心问题：AI 在执行时能获得所需信息吗？
✅ 信息来自明确来源？ ✅ 缺失时有获取方式？ ❌ 信息不可预测？
S
Stability（稳定性）
核心问题：任务的步骤是固定的吗？
✅ 步骤基本一致？ ✅ 目的明确不变？ ❌ 步骤需动态决策？
D
Divisibility（可拆分性）
核心问题：能拆分为 Agent 动作和 LLM 认知吗？
✅ 有确定性步骤？ ✅ 有不确定性步骤？ ✅ 两者可清晰分开？

案例应用

每日站会纪要整理
适合 Skill
代码审查辅助
适合 Skill
修复程序 Bug
不适合
每周数据报表生成
适合 Skill
头脑风暴新产品名字
不适合
解释什么是递归
不适合

RISD 框架的价值

这个框架帮你回答一个核心问题：投入时间写 Skill，到底值不值？

  • Skill 不是万能的，它只适合"重复发生的、流程固定的、信息可得的、能分工协作"的任务
  • 写 Skill 是一种投资——投资你的时间，换取未来的自动化收益
  • RISD 框架就是你的投资决策工具
