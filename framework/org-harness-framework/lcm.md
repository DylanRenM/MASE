# 生命周期定义（LCM）

## 1. 软件开发生命周期模型

本组织采用 **混合生命周期模型**，结合了敏捷和传统方法的优点。

## 2. 生命周期阶段

### 2.1 阶段概览

```
阶段 1: 概念化     → 阶段 2: 需求分析     → 阶段 3: 设计
       ↓                   ↓                   ↓
阶段 4: 实现       → 阶段 5: 测试         → 阶段 6: 部署
       ↓                   ↓                   ↓
阶段 7: 运维支持   → 阶段 8: 退役
```

### 2.2 阶段详细说明

| 阶段 | 名称 | 主要活动 | 关键交付物 |
|------|------|---------|-----------|
| 1 | 概念化 | 市场分析、可行性研究 | 商业案例、可行性报告 |
| 2 | 需求分析 | 需求收集、分析、确认 | 需求规格说明（SRS） |
| 3 | 设计 | 架构设计、详细设计 | 设计文档、API 规范 |
| 4 | 实现 | 编码、单元测试、集成 | 可执行代码、单元测试 |
| 5 | 测试 | 系统测试、验收测试 | 测试报告、缺陷清单 |
| 6 | 部署 | 部署、上线、培训 | 部署文档、用户手册 |
| 7 | 运维支持 | 维护、支持、优化 | 维护记录、性能报告 |
| 8 | 退役 | 数据迁移、系统下线 | 退役报告、数据归档 |

## 3. AI 在各阶段的角色

### 3.1 阶段与 AI 角色映射

| 阶段 | AI 角色 | 具体任务 |
|------|--------|---------|
| 概念化 | 辅助分析 | 市场数据收集、竞品分析 |
| 需求分析 | 辅助整理 | 需求文档生成、用例编写 |
| 设计 | 辅助设计 | 架构建议、代码结构设计 |
| 实现 | 代码生成 | 功能代码生成、测试用例生成 |
| 测试 | 辅助测试 | 测试脚本生成、测试数据生成 |
| 部署 | 辅助部署 | 部署脚本生成、配置文件生成 |
| 运维支持 | 辅助维护 | 日志分析、故障定位 |
| 退役 | 辅助迁移 | 数据迁移方案建议 |

### 3.2 AI 输出要求

- **概念化阶段**：AI 输出的分析报告必须经过人工验证
- **需求分析阶段**：AI 生成的需求文档必须经过利益相关者确认
- **设计阶段**：AI 建议的架构必须经过技术评审
- **实现阶段**：AI 生成的代码必须经过代码评审
- **测试阶段**：AI 生成的测试用例必须覆盖所有需求

## 4. 阶段转换准则

### 4.1 进入下一阶段的条件

| 阶段 | 进入下一阶段的条件 |
|------|-------------------|
| 概念化 | 商业案例获得批准 |
| 需求分析 | SRS 获得评审通过 |
| 设计 | 设计文档获得评审通过 |
| 实现 | 代码完成且通过单元测试 |
| 测试 | 所有测试用例通过 |
| 部署 | 验收测试通过 |
| 运维支持 | 系统稳定运行 30 天 |
| 退役 | 业务迁移完成 |

### 4.2 阶段回退机制

- 如果在某阶段发现严重问题，可以回退到前一阶段
- 回退决策必须经过项目经理批准
- 回退必须记录在项目文档中
