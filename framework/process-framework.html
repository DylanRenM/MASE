第五讲：七步成诗——开发 Skills 的 PROCESS

PROCESS：七个步骤概览

PROCESS 本身就是"流程"的意思——编写 Skill 的过程，就是一个标准流程。

PPlan
RRequirements
OOperations
CCapabilities
EEvaluation
SScript
SSanity
01
P
Plan（明确目标任务）
用一句话说清楚这个 Skill 要完成什么任务。

核心问题：这个 Skill 解决什么问题？用户用一句话触发时，预期得到什么结果？它的边界在哪里？

示例："将每日站会的录音文字，自动整理为包含参会人员、昨日完成、今日计划、阻塞项的结构化会议纪要。"

02
R
Requirements（明确输入信息）
明确 Skill 运行时需要用户提供什么信息。

核心问题：哪些信息是必填的？哪些是可选的？默认值是什么？信息的格式是什么？

技巧：输入越少，Skill 越易用。能用默认值的就不要让用户填。

03
O
Operations（明确步骤与逻辑）
把完成任务需要的步骤全部列出来，并标注每个步骤由谁执行（Agent 还是 LLM）。

核心问题：从输入到输出，经历了哪些步骤？每个步骤的输入输出是什么？

技巧：步骤间用变量名串联，如 {{ cleaned_text }}

04
C
Capabilities（识别工具与权限）
明确 Skill 运行时需要调用什么工具、拥有什么权限。

核心问题：需要调用哪些工具？需要什么权限？敏感操作是否需要用户确认？

重要："禁止什么"比"能做什么"更重要。明确边界防止"过度执行"。

05
E
Evaluation（明确验收标准）
定义"怎样算成功"——输出格式要求、内容完整性、质量标准。

核心问题：输出什么格式？必须包含哪些内容？什么情况下算"成功"？

技巧：验收标准要可验证。不说"输出要清晰"，而说"必须包含参会人员列表"。

06
S
Script（编写 Skill）
将以上所有内容，按照 YAML 格式编写成正式的 Skill 文件。
id: daily-standup-summarizer name: 每日站会纪要整理 inputs: - name: transcript type: string required: true workflow: - id: extract_attendees actor: llm type: prompt input: {system_prompt: "提取参会人员"} output_as: attendees
07
S
Sanity（调试 Skill）
用真实场景测试 Skill，发现问题后迭代修正。

测试清单：正常输入是否正确？边界情况如何处理？约束规则是否生效？错误处理是否符合预期？

迭代流程：测试发现问题 → 修改 YAML → 重新测试 → 重复直到稳定

总结

  • "七步成诗"不是终点，而是起点
  • 第一个 Skill 可能需要一小时，第二个可能半小时，第三个可能十五分钟
  • 当你熟练之后，写 Skill 会像写 Prompt 一样自然
  • 那时你会发现：你不是在"写代码"，而是在"教会 AI 如何帮你工作"
